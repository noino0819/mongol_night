# 도트 스프라이트 제작 가이드 (전 에이전트 필수 준수)

이 프로젝트의 모든 스프라이트 신규 제작·수정 시 반드시 따르는 법칙. 픽셀아트 정석(Lospec·Derek Yu·Pixnote 등) 조사를 16×16·8색 제약에 맞게 압축했다. (조사일: 2026-07-21)

## 0. 재사용 vs 신규 판단
- **의미가 겹치면 재사용** (모닥불→화로 OK). **독립 개념이면 새로 만든다** — 억지 재사용은 안 예쁘고 단서 혼선(방탈출은 특히)까지 만든다.
- 신규 이름은 소문자 영단어, `SPR` 맵에 추가.

## 1. 프로젝트 포맷 (고정)
- **16×16 문자열 맵** → `index.html`의 `const SPR={...}`. 각 스프라이트 = 문자열 16개, 각 줄 **정확히 16자**.
- 문자 = `PAL` 인덱스 `0~7`, `.` = 투명. **이미지 파일 금지.**
- 팔레트 (CSS 변수와 동일, 추가 금지):

| # | HEX | 용도 |
|---|---|---|
| 0 | `#1A1423` | 밤·아웃라인·눈동자 |
| 1 | `#2A2440` | 어두운 남보라(밤하늘·그림자) |
| 2 | `#E8C170` | 모래·금·나무 밝은면 |
| 3 | `#5FA052` | 초원 |
| 4 | `#6FB8D6` | 하늘·물·하닥 |
| 5 | `#D95763` | 불·빨강·전통 문 |
| 6 | `#F4EEE0` | 밀크 화이트(별·종이·하이라이트) |
| 7 | `#8A6044` | 갈색(나무·가죽·동물) |

- 앱 배경이 어두우므로 **밝은 색 실루엣**이 기본. 아웃라인은 원칙적으로 생략(기존 스타일), 필요 시 `0`으로 부분만.

## 2. 제작 법칙 (조사 결과 압축 — 어길 거면 만들지 마)
1. **실루엣 먼저.** 단색으로 형태만 찍어 보고, 실루엣만으로 뭔지 읽히면 그때 색·디테일. 안 읽히면 셰이딩으로 못 살린다.
2. **암시가 디테일을 이긴다.** 16×16에선 점 하나가 눈 전체다. 욕심내서 픽셀 추가 = 노이즈.
3. **부위당 색 1~2개** (바탕+음영). 스프라이트 전체 3~5색. 색이 많으면 뭉개진다.
4. **광원은 좌상단 고정.** 밝은면 좌상, 음영 우하. 전 스프라이트 일관.
5. **필로우 셰이딩 금지** — 외곽선 따라 안쪽으로 고르게 어두워지는 것. 광원 무시라 흐릿해 보인다.
6. **밴딩 금지** — 명도 다른 색이 외곽선과 평행하게 줄지어 띠처럼 보이는 것. 덩어리로 끊어라.
7. **고아 픽셀 금지** — 혼자 뜬 1픽셀은 (의도된 텍스처가 아니면) 노이즈. 2×1 이상 덩어리로.
8. **청키 픽셀 규칙** — 팔·다리·기둥 등 돌출부는 1픽셀 두께 금지, 최소 2픽셀. 가늘면 부서져 보인다.
9. **직선 계단(jaggies) 최소화** — 대각선은 1-2-1 같은 규칙적 계단으로. 불규칙 계단은 지저분하다.
10. **디더링은 원칙적으로 안 쓴다** (16×16에선 노이즈). 질감은 색 덩어리로.
11. **게임 단서와 그림 일치** — 퍼즐이 "별 7개"를 세게 하면 스프라이트에 별이 정확히 7개(단서색과 배경 장식색 구분). 그림이 곧 UI다.

## 3. 검증 (제작 후 필수)
```bash
node tools/sprite-lint.mjs   # SPR 전 스프라이트: 16행×16자·유효문자(0-7,.)·고아픽셀 경고
```
- 육안 체크: scale 3~4로 렌더해 실루엣 읽히는지, 광원 좌상인지.
