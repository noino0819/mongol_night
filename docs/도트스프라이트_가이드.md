# 도트 스프라이트 제작 가이드 (전 에이전트 필수 준수)

이 프로젝트의 모든 스프라이트 신규 제작·수정 시 반드시 따르는 법칙. 픽셀아트 정석(Lospec·Derek Yu·Pixnote 등) 조사를 16×16·8색 제약에 맞게 압축했다. (조사일: 2026-07-21)

## 0. 재사용 vs 신규 판단
- **의미가 겹치면 재사용** (모닥불→화로 OK). **독립 개념이면 새로 만든다** — 억지 재사용은 안 예쁘고 단서 혼선(방탈출은 특히)까지 만든다.
- 신규 이름은 소문자 영단어, `SPR` 맵에 추가.

## 1. 프로젝트 포맷 (고정)
- **16×16 문자열 맵**. 각 스프라이트 = 문자열 16개, 각 줄 **정확히 16자**.
- **등록 위치 (병렬 세션 충돌 방지 — 필수)**: index.html은 스프라이트 작업에서 절대 건드리지 않는다.
  - 특정 게임 전용 → **그 게임의 `games/*.js` 상단**에서 등록:
    ```js
    /*SPR_BEGIN*/
    snAddSprites({ 이름:[ ...16행... ], });
    /*SPR_END*/
    ```
  - 여러 게임 공용(동물·앱셸·주사위 등) → **`assets/sprites.js`의 `SPR`에 추가** (index.html이 `<script src>`로 로드).
  여러 세션이 index.html 인라인 SPR을 동시에 고치다 유실 사고가 났던 구조라(2026-07 er 11종 유실), 이 분리가 기본이다. 이름이 겹치면 sprite-lint가 에러로 잡는다.
- 문자 = `PAL` 인덱스 `0~7`, `.` = 투명. **이미지 파일 금지.**
- 팔레트 (CSS 변수와 동일, 추가 금지):

| # | HEX | 용도 |
|---|---|---|
| 0 | `#1A1423` | 밤·아웃라인·눈동자 |
| 1 | `#2A2440` | 어두운 남보라(밤하늘·그림자) |
| 2 | `#E8C170` | 모래·금·나무 밝은면 |
| 3 | `#5FA052` | 초원 |
| 4 | `#6FB8D6` | 하늘·물·하닥 |
| 5 | `#D95763` | 불·빨강·전통 문 |
| 6 | `#F4EEE0` | 밀크 화이트(별·종이·하이라이트) |
| 7 | `#8A6044` | 갈색(나무·가죽·동물) |

- 앱 배경이 어두우므로 **밝은 색 실루엣**이 기본. 아웃라인은 원칙적으로 생략(기존 스타일), 필요 시 `0`으로 부분만.

## 2. 제작 법칙 (조사 결과 압축 — 어길 거면 만들지 마)
1. **실루엣 먼저.** 단색으로 형태만 찍어 보고, 실루엣만으로 뭔지 읽히면 그때 색·디테일. 안 읽히면 셰이딩으로 못 살린다.
2. **암시가 디테일을 이긴다.** 16×16에선 점 하나가 눈 전체다. 욕심내서 픽셀 추가 = 노이즈.
3. **부위당 색 1~2개** (바탕+음영). 스프라이트 전체 3~5색. 색이 많으면 뭉개진다.
4. **광원은 좌상단 고정.** 밝은면 좌상, 음영 우하. 전 스프라이트 일관.
5. **필로우 셰이딩 금지** — 외곽선 따라 안쪽으로 고르게 어두워지는 것. 광원 무시라 흐릿해 보인다.
6. **밴딩 금지** — 명도 다른 색이 외곽선과 평행하게 줄지어 띠처럼 보이는 것. 덩어리로 끊어라.
7. **고아 픽셀 금지** — 혼자 뜬 1픽셀은 (의도된 텍스처가 아니면) 노이즈. 2×1 이상 덩어리로.
8. **청키 픽셀 규칙** — 팔·다리·기둥 등 돌출부는 1픽셀 두께 금지, 최소 2픽셀. 가늘면 부서져 보인다.
9. **직선 계단(jaggies) 최소화** — 대각선은 1-2-1 같은 규칙적 계단으로. 불규칙 계단은 지저분하다.
10. **디더링은 원칙적으로 안 쓴다** (16×16에선 노이즈). 질감은 색 덩어리로.
11. **게임 단서와 그림 일치** — 퍼즐이 "별 7개"를 세게 하면 스프라이트에 별이 정확히 7개(단서색과 배경 장식색 구분). 그림이 곧 UI다.
12. **대칭 여부는 사물의 본성을 따른다 (눌림 방지).** 비대칭 자체는 자유다 — 동물·불꽃·하닥처럼 개념이 비대칭인 건 억지로 대칭 맞추지 마라(생동감이 죽는다). 단, 기어·주사위·하트·종처럼 **상식적으로 대칭이어야 하는 사물**은 **실루엣**(색 말고 채움/투명만)을 축에 맞춘다. 특히 방사형(기어·별·폭발)은 상하 돌출부와 좌우 돌출부의 **모양·두께가 다르면 눌려 보인다** — 돌출부는 전부 같은 크기 블록(2×2 등)으로 통일. 16그리드 중심축은 7|8 사이(셀 7.5)라 손으로 어긋나기 쉬우니, 헷갈리면 스크립트로 생성·검증하라. 대칭이어야 하는 스프라이트만 `tools/sprite-lint.mjs`의 `SYM` 맵에 등록해 자동 검사를 건다(H=좌우 · V=상하 · R=90° 회전, 등록제 — 비대칭 스프라이트는 등록하지 않으면 검사 대상이 아니다). 광원 셰이딩(색)은 비대칭이어도 됨 — 검사도 실루엣만 본다. (사례: v2.6.0 gear가 상하 이빨만 테이퍼라 눌려 보임 → v2.6.1 수정, R 검사 도입 계기)

## 3. 검증 (제작 후 필수)
```bash
node tools/sprite-lint.mjs   # SPR 전 스프라이트: 16행×16자·유효문자(0-7,.)·고아픽셀 경고·SYM 대칭(에러)
```
- 대칭이어야 하는 오브젝트(2장 12번)만 lint의 `SYM` 맵에 이름과 축(H/V/R)을 등록 — 등록제라 비대칭 스프라이트엔 아무 제약 없다.
- 육안 체크: scale 3~4로 렌더해 실루엣 읽히는지, 광원 좌상인지.
